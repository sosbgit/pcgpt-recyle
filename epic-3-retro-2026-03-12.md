## Epic Summary

Six stories delivered the full ADS-B ingestion pipeline and Signal Desk feed health visibility in a single execution session. The pipeline runs from HTTP fetch → normalization → Valkey pub/sub → PostgreSQL persistence, with a passive feed health API and live Signal Desk indicators closing the loop.

| Story | Scope |
|---|---|

## 1. What Went Well

**Architecture held without bending.** Every service boundary stayed clean: normalizer is pure Python with no I/O, Valkey publish excludes raw_payload, geography_entity_id is always NULL from the ingestor, compliance guard is called before any I/O. No boundary violations surfaced during implementation.

**Layered story sequencing paid off.** Each story built on the previous one's seams without touching completed files unnecessarily. The `__init__` signature extension pattern (`config → config, settings → config, settings, redis_client → config, settings, redis_client, db_writer`) allowed backward-compatible growth with no test regressions across stories.

**Passive feed health model worked first try.** The SETEX/TTL approach — ingestor writes `feed:health:opensky` with 30s TTL every 15s, API reads TTL only, absence IS the stale signal — was confirmed end-to-end: live/stale transitions in the UI within ~15s of ingestor stop/start, no page reload required.

**Test discipline was strong.** Pure-Python unit tests throughout (no real DB/Valkey/network required for `pytest`). Test counts: 43 tests (3.1), ~60+ tests (3.2 adds Groups A–D), Groups H/V (3.3), Groups W (3.4), Groups F (3.5). All passed on smoke test for each story.

**Real-scale smoke tests were meaningful.** Live OpenSky runs hitting 8,000–10,000 aircraft per cycle confirmed the pipeline under real load, not just synthetic test data. Compliance gate (`raw_adsb` count = 0, `entity_states` count = N) verified in production-like conditions.

**React Query deduplication worked as designed.** `ChromeBar.tsx` and `SignalDeskPage.tsx` both calling `useFeedHealth()` independently resulted in a single network request per poll cycle — confirmed in smoke test.

**No scope creep.** Every story held its deferred items list and did not drift into adjacent stories.

## 2. What Was Painful or Error-Prone

**Windows packaging mismatch (3.1).** `panoptes-shared @ file://../../shared` in `pyproject.toml` failed on Windows during smoke test. Had to change to bare `panoptes-shared` name. This was discovered at smoke test time, not earlier. The story spec originally specified the wrong form.

**Pydantic v1-style `class Config:` carried forward through the entire epic.** `IngestorSettings` was written with `class Config:` (v1 pattern) in 3.1 and the deprecation warning was noted as non-blocking in every story from 3.2 through 3.4. It was never fixed. This accumulated as explicit carry-forward debt.

**Ctrl+C shutdown noise.** `asyncio.gather` teardown produces `CancelledError` during sleep + `KeyboardInterrupt` on exit — logged as "non-blocking" in every story from 3.3 onward. No signal handler was added. This is noise every time the ingestor is stopped locally.

**Code review catches were systematic, not incidental.** Each story required a post-implementation code review pass that caught real issues:
- 3.3: `published_count` was incrementing per call, not per successful publish. Required a targeted fix.
- 3.3: One-time "no-redis" debug log was emitting repeatedly per cycle. Required a fix.
- 3.4 (F1): Duplicate broad `except Exception` block in `writer.py` masked the intentional `SQLAlchemyError` handler.
- 3.4 (F2): Stale `run()` docstring in `opensky.py` still referenced old behavior.
- 3.5: `redis.asyncio.Redis` type annotation was loose in the feeds router.

These were all caught before shipping, but the pattern suggests stories consistently needed a second pass.

**PostGIS coordinate order is a silent-corruption trap.** `WKTElement("POINT(longitude latitude)")` — longitude first — is not intuitive and is not checked by the type system or the ORM. A swap would produce geometrically valid points in the wrong hemisphere with no error. The spec called it out, but it relies on the dev reading carefully.

**`__init__` signature kept growing.** By 3.4, `OpenSkyAdapter.__init__` had four positional-optional parameters: `config, settings=None, redis_client=None, db_writer=None`. Not a bug, but this pattern will continue to grow in Epic 4+ if not managed.

## 3. Technical and Process Lessons to Carry Forward into Epic 4

**1. Migrate `IngestorSettings` to Pydantic v2 `model_config` before 4.1.**
The `class Config:` pattern is v1 and emits a deprecation warning on every startup. Epic 4 adds the processor service — that service's settings should use `model_config` from the start, and 3.x debt should be cleared in the first processor story or a short housekeeping task.

**2. Add a SIGTERM/SIGINT shutdown handler before Epic 4 stories use the ingestor.**
`asyncio.gather` + Ctrl+C noise is acceptable in dev but will become a logging problem in any supervisor context (Docker, systemd). A clean shutdown path (`signal.signal(SIGTERM, handler)` → cancel tasks) should be added. This is a standalone housekeeping item, not a story-level concern.

**3. Document `POINT(lon lat)` coordinate order as a project-wide hard rule in `project-context.md`.**
Every service that writes PostGIS geometry must use longitude-first WKT. The processor (Epic 4) will not write geometry directly, but Epic 8 (Spatial View) will. Capturing this now prevents a silent data corruption incident later.

**4. Validate Windows install sequence in story specs at the pyproject.toml level.**
When a story adds a new package that references a local package (e.g., `panoptes-shared`), the spec should verify the install form is Windows-safe at spec time, not smoke-test time. Bare package name (`panoptes-shared`) is the confirmed pattern — state it explicitly in dependency tables for future ingestor/service stories.

**5. Counter variables in loop bodies should be defined at spec time.**
Three stories required post-review fixes to counter tracking logic (`published_count`, `written_count`). Story specs for Epic 4 loop-based work should explicitly define counter semantics (increment only on return value = `True`, not on call).

**6. One-time init logs vs. per-record logs is a design decision, not a code review catch.**
The pattern (`should_write_raw=False` warning logged once at init, never per-record) was correct but the first implementation got it wrong in subtle ways. Future stories with loop-based I/O should make this explicit in the AC: "emit once at init, never per-record."

**7. Grow the `OpenSkyAdapter.__init__` signature carefully.**
By the end of Epic 3, the constructor takes four parameters. Consider whether Epic 4's processor adapter should use a dataclass or config object instead of continuing to grow positional optionals.

## 4. Recurring Review / Smoke-Test Issues That Should Change How Stories Are Executed

**Counter correctness belongs in AC, not in code review.**
Stories 3.3 and 3.4 both had counter-tracking bugs (incrementing on call vs. on success). Going forward: any story that adds a cycle-level counter must include an explicit AC test case that verifies increment-on-success-only semantics (e.g., "if write() returns False, written_count does NOT increment").

**Docstring correctness is hard to spec, easy to break.**
Two stories (3.4 F2) had stale docstrings that referenced superseded behavior. Going forward: after each story updates a `run()` loop or `__init__`, the dev agent should do a targeted pass on all docstrings in modified files before marking the story review-ready.

**Type annotation completeness should be a smoke-test checklist item.**
3.5 had a loose type annotation on `redis.asyncio.Redis` caught in review. Going forward, every new function signature in FastAPI routers and async adapters should have complete type annotations — this is checkable with `mypy --strict` or `pyright`. Consider adding a `pyright` or `mypy` check to the story's "build verification" AC.

**Windows local dev install sequence should be in the story's smoke-test AC.**
The 3.1 packaging issue was found at smoke test. Future stories that touch `pyproject.toml` should include an explicit smoke-test step: `pip install -e shared/ -e "services/<service>[test]"` on Windows (no `file://` path references, bare package names only).

## 5. Concrete Recommendations for Epic 4

**Before starting 4.1 (Processor Service Base):**

1. **Housekeeping task: fix `IngestorSettings` Pydantic v2 migration.** Change `class Config:` to `model_config = SettingsConfigDict(...)`. Single-file change, no story needed — carry it as a task on 4.1 pre-work or as a standalone sub-task.

2. **Housekeeping task: add SIGTERM handler to `services/ingestor/ingestor/main.py`.** Single-function addition; prevents noisy shutdown in any future container context.

3. **Add `project-context.md` rule: PostGIS coordinate order.** Add one line under "Service Boundary Write Rules" or a new "Spatial Conventions" section: `WKTElement("POINT({longitude} {latitude})", srid=4326)` — longitude always first.

**For 4.1 story spec:**

4. **Processor should follow the ingestor's pattern strictly** — base class, settings via `model_config`, startup compliance/validation, no DB/Valkey access in the base class, tests before wiring. The ingestor pattern is proven; replicate it.

5. **Valkey subscriber pattern:** The processor will use `PSUBSCRIBE entity:adsb:*`. Spec the subscription lifecycle explicitly — including reconnect on disconnect — before writing code, the same way 3.2 specified the fetch loop backoff.

6. **Emit a processor heartbeat key** (`proc:health:processor`, TTL=30s, 15s interval) from day one in the processor. Epic 5's Signal Desk needs to surface processor liveness the same way it surfaces ingestor liveness — building it in at 4.1 costs nothing and avoids a retrofit story later.

7. **`geography_entity_id` assignment:** The first processor story must spec exactly when and how `entity_states.geography_entity_id` is populated via geospatial watch-matching. The NULL-from-ingestor rule is in place — the processor must set it or leave it NULL with clear semantics.

8. **Keep `signal-first, homelab-first` context constraints explicit in every 4.x story.** No prod/nginx drift. OpenSky remains personal/homelab only with `storage_allowed=True, terms_status="under_review"` — the compliance posture is NOT changed during Epic 4. Any processor story that needs to read raw_adsb must handle the `raw_adsb` table being empty gracefully.
